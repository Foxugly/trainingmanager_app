# S2-b — Events list + detail + RSVP Implementation Plan

> Executes inline (autonomous). Hand-written DTOs (codegen deferred). ROTI / rounds-exercises / attachments → S2-c.

**Goal:** Athlete can list events, open an event, see its details (honoring `vis_distance`/`vis_goal`), and set their RSVP (going/maybe/not_going). Reached from a "Events" entry on the dashboard.

**Contracts (schema, all `jwtAuth`):**
- `GET /api/v1/events/` → **`PaginatedEventList`** `{count, next, previous, results:[Event]}` (PAGINATED wrapper — diverges from the fleet bare-array convention; read `.results`). Optional query: `date__gte`, `date__lte`, `page`, `page_size`, `search`, `ordering`.
- `GET /api/v1/events/{id}/` → `Event` (bare).
- `GET /api/v1/events/{event_pk}/rsvp/` → `RsvpSummary`; `PUT .../rsvp/` body `RsvpUpsertRequest {status: going|maybe|not_going}` → `RsvpSummary`. 400 invalid status, 403 RSVP disabled / not an athlete-member.
- `Event` fields used: id, name, goal?, location, equipment, color?, date?, hour_start?, hour_end?, total (distance), place?(PlaceMinimal{id,name,address}), refer_program(ProgramMinimal{id,name}), vis_distance/vis_goal/vis_rounds (VisibilityMode `always|after|never`), debrief. Everything else ignored via ignoreUnknownKeys.
- `RsvpSummary {counts:RsvpCounts{going,maybe,not_going,no_response}, total_members, my_status?(going|maybe|not_going|null), by_member[]}`.
